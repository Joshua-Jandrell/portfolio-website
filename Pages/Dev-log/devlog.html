    Going back to the 'back' button.

    The back button was never well placed on my wireframes and has always felt like an award appendage cluttering up the site. I'm still not 100% sure where to put it, but at least I've thought things through.

    What doess the back button do?

    The back button allows the user to go back to where they game from. I'm renaming it the return button because the fits the “programming” theme better. (Return is the command given to leave a function).

    What's the issue with it?

    The back buttons on the blog and dev-log pages are a bit of a mess. They float around haphazardly feeling like clutter wherever they go.

    I've moved most navigation and index features to the right, which is great, but it means that there is even less space for the button to naturally rest.

    I was considering putting it on the nav bar - but I want the nave bar to have the same fields throughout the site to give a sense of consistency. The nav-bar is also quite cluttered as it is.

    Can't the user just navigate with the nav bar? Why is return so important?

    Currently it's not doing that much. But once I start adding some JS the return button will send the user back to their previous page, wherever that was. This makes navigating the site easier (which is never a bad thing). So, no. No, I do not want to do away with the return button.

    To be honest I'm not sure where it should go. I could throw it onto the right-hand side, but it would look weird and lonely there. Besides, the website is in English which sets a natural left-to right convention. Returning or going back can be seen as leftwards movement, so putting a return button on the right may seem unintuitive.

    I'm going to be reformatting it to look like a piece of code and throw it under the title. I'll pop it again at the bottom of the article. Logically these are the two points where a reader is most likely to decide they want to go back. Either they haven't started and decide they'd rather look at something else, or they finished reading and want to go back.

    With this in mind I'll probably be floating a return button above the index section and again at the bottom of the article. Being above the index will put the button in the comfortable top left corner - where one expects a back button to be. The one at the bottom of the page let's users return when they are finished reading. Is this overkill? Perhaps a back at the end is goo enough. Is there a point in asking someone to leave when they just arrived?

    Possibly. I'll have to wait and see. I think I'll throw it in at below the index (at least for now). This groups it with naiation which seems logical. I can always re-move it later…

    Pseudo square brackets: what, how, why?


    Square brackets have been a recurring motif on my sight because they invoke ideas of theology and programming.
    (They also look cool!)
    . But using inline text elements to represent them is tedious and creates clutter for screen reader users: there must be a better way.

    Why do you want these square brackets anyway?

    Decoration. This site isn't exactly putting on a sceptical with pretty images and fancy graphics. Of course, this is by design - but having no decoration at all is boring. The brackets add a bit of intrigue and variety while acting as a motif. They give the site a distinct aesthetic style with should make it more memorable to visitors. Furthermore, it ensures that all pages on the site (I know, this sounds like a horribly mixed metaphor…) are look like parts of a common whole; even when their layout and content is varied.

    So? Why not are semi colons, ampersands, or OTHER BRACKETS - these are used in code too!

    Well, yes. But square ones are the best! (In my case anyway).

    Firstly, square brackets are often used for indexing in programming languages. And this is how I am using brackets on my site. They indicate tilts and section: items that would appear in and index. If my site were some sort of dictionary class, this is exactly how a under might expect to retrieve the section associated with a title. Tying a functional meaning to the elements is important because it reinforces the connection to programming (and shows that I know what I'm doing).

    Other coding syntax elements lack his meaning. Semi-colons are just line endings and curly brackets often just contain sections of code. (As for good old round brackets, their meaning is already well defined as an element of English punctuation. Not not just meaningless, they're but confusing too!)

    I also like square brackets because they appear strong and confident. Most containers on this site have crisp, sharp boarders with unrounded edges - square braces fit comfortably into this astatic. (Although I must confess square braces may have also informed this athletic)

    So what's wrong with them?

    If you are viewing this site on a monitor, they probably look the same. But if you view them with a screen reader one is haven then the other is hell. (And not becuase of that top and bottom thing.)

    Exactly! That's what happens if decorative square brackets are added as inline characters. Like I did with the first div (and most of the early headers on this site).

    Take a look at the code for the first one to see what I mean: 

                        
  <style>
    .square-bracket-bad {
      font-weight: bold;
      color: rgb(47, 177, 203);
    }
    .square-bracket-bad .inner {
      font-weight: normal;
      color: black;
      margin: 0 10px;
    }
  </style>

  <div class="square-bracket-bad">
    [<span class="inner">Something in brackets</span>]
  </div>

    Oh dear. That's a bummer... I'm guessing the second div is different...

    How do you fix this>

    Magic. (Actually CSS, which is basically the same thing.)

    I made used of the ::before and ::after CSS pseudo elements. These elements render (you'll never guess this) before and after their parent element. Think of them like the bullet points on a list. They are pretty neat and simple to use.

    Just look at the code for the lower div:

                        
  <style>
   .square-bracket-good {
      font-weight: normal;
      color: black;
    }
    .square-bracket-good::before {
      content: "[";
      font-weight: bold;
      color: rgb(47, 177, 203);
      margin-right: 10px;
    }
    .square-bracket-good::after {
     content: "]";
     font-weight: bold;
     color: rgb(47, 177, 203);
     margin-left: 10px;
  }
  </style>

  <div class="square-bracket-good">Something in brackets</div>

    Did you notice the lack of braces? Well screen reader users noticed that too! So did web crawlers and anything else that is interfacing directly with the code on my site. Removing square brackets and other decretive clutter makes the code more sematic and machine readable.

    Pseudo elements aren't real. They can't hurt you. (Pseudo literally means fake!) These elements are defined entirely with scc styling which means that, like all good children in the Middle Ages, they can be seen but not heard.

    Another great thing about the pseudo element approach is the flexibility it gives me. Let's say I get hooked on C++ and decide that titles aren't really indexes _ there're pointers. So, I want to change all of the []'s into '’s. If I need to change the characters next to every single heading, I'd probably give up or pass out from boredom. Fortunately, with pseudo elements this is as simple as changing two lines of CSS . If the style s applied to a class, this can allow innumerable html elements to be reformatted.
