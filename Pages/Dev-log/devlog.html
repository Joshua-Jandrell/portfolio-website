    Pseudo square brackets: what, how, why?


    Square brackets have been a recurring motif on my sight because they invoke ideas of theology and programming.
    (They also look cool!)
    . But using inline text elements to represent them is tedious and creates clutter for screen reader users: there must be a better way.

    Why do you want these square brackets anyway?

    Decoration. This site isn't exactly putting on a sceptical with pretty images and fancy graphics. Of course, this is by design - but having no decoration at all is boring. The brackets add a bit of intrigue and variety while acting as a motif. They give the site a distinct aesthetic style with should make it more memorable to visitors. Furthermore, it ensures that all pages on the site (I know, this sounds like a horribly mixed metaphor…) are look like parts of a common whole; even when their layout and content is varied.

    So? Why not are semi colons, ampersands, or OTHER BRACKETS - these are used in code too!

    Well, yes. But square ones are the best! (In my case anyway).

    Firstly, square brackets are often used for indexing in programming languages. And this is how I am using brackets on my site. They indicate tilts and section: items that would appear in and index. If my site were some sort of dictionary class, this is exactly how a under might expect to retrieve the section associated with a title. Tying a functional meaning to the elements is important because it reinforces the connection to programming (and shows that I know what I'm doing).

    Other coding syntax elements lack his meaning. Semi-colons are just line endings and curly brackets often just contain sections of code. (As for good old round brackets, their meaning is already well defined as an element of English punctuation. Not not just meaningless, they're but confusing too!)

    I also like square brackets because they appear strong and confident. Most containers on this site have crisp, sharp boarders with unrounded edges - square braces fit comfortably into this astatic. (Although I must confess square braces may have also informed this athletic)

    Exactly! That's what happens if decorative square brackets are added as inline characters. Like I did with the first div (and most of the early headers on this site).

    Take a look at the code for the first one to see what I mean: 

                        
  <style>
    .square-bracket-bad {
      font-weight: bold;
      color: rgb(47, 177, 203);
    }
    .square-bracket-bad .inner {
      font-weight: normal;
      color: black;
      margin: 0 10px;
    }
  </style>

  <div class="square-bracket-bad">
    [<span class="inner">Something in brackets</span>]
  </div>

    Oh dear. That's a bummer... I'm guessing the second div is different...

    How do you fix this>

    Magic. (Actually CSS, which is basically the same thing.)

    I made used of the ::before and ::after CSS pseudo elements. These elements render (you'll never guess this) before and after their parent element. Think of them like the bullet points on a list. They are pretty neat and simple to use.

    Just look at the code for the lower div:

                        
  <style>
   .square-bracket-good {
      font-weight: normal;
      color: black;
    }
    .square-bracket-good::before {
      content: "[";
      font-weight: bold;
      color: rgb(47, 177, 203);
      margin-right: 10px;
    }
    .square-bracket-good::after {
     content: "]";
     font-weight: bold;
     color: rgb(47, 177, 203);
     margin-left: 10px;
  }
  </style>

  <div class="square-bracket-good">Something in brackets</div>
